# Support Playbook (§11.9 / §15)

Руководство дежурного поддержки: SLA, типовые обращения, эскалации, инструменты.

## SLA

- Первый ответ в рабочие часы: ≤ 2 часа (§1.4).
- Критические инциденты (оплата/массовый сбой генерации): реакция ≤ 30 минут.
- Метрики нагрузки и первого ответа: web-admin → «Статистика поддержки» (`/admin/support/stats`).

## Каналы и инструменты

- Обращения пользователей: экран FAQ/Поддержка (mobile) и `/support` (API `/api/v1/support`).
- Панель агента: web-admin (через VPN + 2FA).
- Алерты: Telegram-канал (очередь, воркеры, платежи, GPU).
- Наблюдаемость: Grafana (дашборды очереди/качества/платежей).

## Типовые сценарии

### 1. «Модель не сгенерировалась / ошибка»

1. Найти заказ по `#id`/`task_uuid` в web-admin.
2. Проверить статус и `quality_report.json` (quality-gate по геометрии/сегментации).
3. Если quality-gate провалил (`empty_geometry`/`low_poly`) — предложить пересъёмку 12 ракурсов.
4. При сбое воркера — проверить очередь/heartbeat; эскалация приводит к авто-возврату (§4.2).

### 2. «Оплата прошла, а баланс/заказ не обновился»

1. Проверить YooKassa webhook (подпись/IP), логи вебхуков и streak-алерты.
2. Проверить pending-платежи; при «зависании» — refresh статуса по API YooKassa.
3. При двойном списании — оформить возврат; зафиксировать в аудите.

### 3. «Заблокировано как NSFW»

1. Подтвердить корректность блокировки (порог `NSFW_THRESHOLD`).
2. Если ложное срабатывание — разблокировать и пересчитать; средства возвращаются (block→refund→unblock).

### 4. «Не приходят уведомления»

1. Проверить настройки уведомлений пользователя (push/email тумблеры).
2. Проверить доставку push и email-fallback (5 минут, если push не открыт).
3. Проверить FCM-конфигурацию при массовой проблеме.

### 5. «Проблемы со входом / 2FA»

1. Проверить статус аккаунта (blocked/verified).
2. 2FA: пользователь может отключить сам (кроме владельца компании — обязательна).
3. Скомпрометирована сессия — пользователь завершает чужие сессии в настройках; при необходимости — сброс токенов.

### 6. B2B-обращения

1. Проверить компанию: лимиты участников, API-ключи, индивидуальные цены (web-admin → карточка компании).
2. Приглашения: список/отзыв в разделе приглашений.
3. ERP-вебхуки: проверить fail-streak алерты.

## Эскалация

- L1 (агент): типовые обращения, ответы по FAQ, возвраты по регламенту.
- L2 (инженер on-call): сбои воркеров, БД/Redis/MinIO, платежи, массовые инциденты.
- L3 (владелец/архитектор): инциденты безопасности, утечки ПД, критические финансовые расхождения.

Порядок: L1 → при превышении SLA/критичности → L2 (Telegram on-call) → L3.

## Чек-лист закрытия обращения

- [ ] Причина установлена и устранена/обойдена.
- [ ] Пользователь уведомлён (ответ по SLA).
- [ ] Действия зафиксированы в аудите (возвраты/разблокировки).
- [ ] При системной проблеме — заведён тикет инженерии и/или обновлён FAQ.
