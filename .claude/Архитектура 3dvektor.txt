1. Выделенный сервер (или VPS/Облачная ВМ) — «Оркестратор и Фронтенд»

Это главный «мозг» системы, доступный извне (или через прокси). На нём работают все пользовательские интерфейсы и бизнес-логика, не требующая тяжелых GPU-вычислений и долгого хранения больших файлов.
Контейнеризация: Docker + Docker Compose.
Компоненты:
Tailscale сервер для организации приватной сети между узлами и присвоении каждому узлу внутреннего приватного статичного ip адреса (администрируется из панели руководителя) 
API Gateway / Оркестратор (FastAPI): Основной бэкенд. Обрабатывает запросы от мобильных приложений и веб-сайта, управляет JWT, очередями задач, вебхуками (ЮKassa), бизнес-логикой (промокоды, роли, налоги).
Веб-панель владельца сервиса + Пнели технической поддержки: React/Streamlit-приложение (доступно только через HTTPS+2FA).
Веб-сайт для селлеров (личный кабинет): React/Next.js (доступен пользователям для просмотра баланса, истории генераций, доступными/использованными промокодами, управления командой).
Celery (Worker & Beat): Фоновые задачи (отправка писем, напоминаний, синхронизация очереди с БД, генерация отчетов).
Nginx: Reverse-proxy для статики и маршрутизации HTTPS-трафика.


2. Кластер хранения данных (Storage HA Cluster) — 2 отдельных физических сервера

Согласно разделу 22, вся персистентная нагрузка (БД, файлы, логи) вынесена с оркестратора на два отдельных физических сервера (Домашний ПК-1 (Server1) и Домашний ПК-2 (Server2)). Это делается для того, чтобы даже при падении одного ПК данные сохранились, а сервис переключился на реплику.
Сетевой слой: Оба узла, Воркеры и оркестратор объединены в закрытую VPN-сеть через Tailscale. Оркестратор обращается к ним по их Tailscale-IP (или через VIP).


Узел А (Primary / Master) — основной активный узел

На этом сервере все сервисы работают в режиме «Master» и принимают запросы на запись от Оркестратора.
PostgreSQL (Patroni): Главная реляционная БД (пользователи, заказы, компании, метаданные моделей, финансы).
Redis (Master): Хранилище очередей задач, heartbeat-воркеров, кэш сессий. Настроен с AOF-персистентностью.
ClickHouse (Primary): Хранилище для агрегированных метрик и долгосрочных логов (аналитика).
MinIO (S3-совместимый Primary): Хранилище файлов (ZIP-архивы с фото, готовые .glb-модели, видео, резервные копии БД). Все данные, загруженные сюда, асинхронно реплицируются на Узел Б.


Узел Б (Standby / Replica) — горячий резервный узел

Полностью дублирует конфигурацию Узла А, но работает в режиме реплики. Его задача — мгновенно подхватить нагрузку, если Узел А упал.
PostgreSQL (Patroni Replica): Потоковая репликация. При падении Узла А Patroni автоматически повышает этот узел до Primary (используется кворум-арбитр на сервере Оркестратора для защиты от Split-Brain).
Redis (Sentinel Replica): Реплика Master’а. При падении Master’а Sentinel продвигает эту реплику.
ClickHouse (Replica): Используется движок ReplicatedMergeTree — синхронизация партиций происходит автоматически.
MinIO (Replica): Асинхронная репликация бакетов (MinIO Bucket Replication). Если Узел А «лёг», оркестратор переключает генерацию presigned URL на этот узел.
Особенности связи: Оркестратор не «дергает» оба узла напрямую. Используется Tailscale VIP (Keepalived) или Application-level Fallback (оркестратор пробует подключиться к Узлу А(Server1), а если нет ответа — к Узлу Б(Server2)). При возвращении Узла А в сеть запускается механизм «догоняющей синхронизации» (Catch-up): PostgreSQL стягивает WAL-логи, MinIO сверяет хэши и копирует недостающие файлы с Узла Б.


3. Вычислительные узлы (Воркеры) — неограниченное количество машин

Это самый распределенный слой. Каждый воркер — это отдельный физический или виртуальный сервер (домашний ПК владельца или арендованный облачный инстанс с GPU (который с помощью скрипта может "в два клика" разворачиваться и запускаться/останавливаться у облачного gpu провайдера)).
Локация: Могут находиться где угодно (дома у владельца, в дата-центре, в облаке), но обязательно подключены к Tailscale-сети (основной канал) или используют резервный WSS (WebSocket Secure) для исходящего подключения, чтобы пробивать NAT.
Компоненты (упакованы в единый Docker-образ):
TRELLIS + PyTorch: Ядро генерации 3D.
Агент воркера (Python): Подключается к Оркестратору по WebSocket, слушает команды (task, stop), отправляет heartbeat (каждые 5 сек) и метрики (температура GPU, загрузка).
Скрипты постобработки: Удаление фона (DeepLabV3+ + SAM), ретопология, внедрение DWT-водяных знаков (только в diffuse), Draco-сжатие.
Принцип работы: Воркер не хранит данные постоянно. Он получает ссылку на ZIP из MinIO, скачивает его во временную папку, генерирует модель, загружает результат обратно в MinIO-кластер, отдает presigned URL оркестратору и очищает temp-папку.
